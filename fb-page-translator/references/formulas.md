# 三模式公式詳細

> 翻譯員 voice 三模式公式：F-TOOL-DIVE（工具探討）/ F-DAILY-NEWS（新聞短評）/ F-WEEK-RECAP（本週大事）。
>
> **使用前必讀**：`../style_profile.md` 是最高優先 — 任何草稿都必須先通過 `voice_lock.md` 的 8 條檢查。

---

## 🔧 F-TOOL-DIVE：工具探討（一三五用）

**核心鉤子**：hands-on 心得 + 工具實際應用 + 戰略含義
**最吃的平台**：FB Page + LinkedIn
**字數**：350-400
**用途**：每週 Mon/Wed/Fri 21:00 — 用 hands-on 心得帶出 AI 工具的應用啟示

### 結構（4 段）

```
[段 1 — 工具名 + 我用它做了什麼 + 戰略點題]
**最近我用 [工具名] 來 [具體任務]**。它的核心能力是 [一句工具介紹]。對企業導入 AI 的人來說，這種工具代表 [一句戰略含義]。

[段 2 — 用之前 vs 用之後（量化差異）]
過去做這件事我會 [舊做法 + 痛點]。現在 [新做法 + 量化改變]。我自己這半年 hands-on 的觀察是：[補充心得]。

[段 3 — 普通人 / 企業可以怎麼用]
從經營管理的角度看，[行業 / 場景] 也可以拿來做 [實際應用]。最直接的價值是 [可量化的影響]。[補充戰略觀點]。

[段尾 — 什麼時候該用 / 不該用]
[一句使用時機建議或長期觀察]。

—

資料來源：
[Source name] — URL
```

### 題材池

**你的 GitHub repos**（任何公開 repo 都是素材）：
- 自己寫的 AI 工具 / agent 框架 / pipeline
- 練習用的 AI 專案

**外部 AI 工具**（近期使用過）：
- Claude Code / Cursor / Antigravity / Codex
- NotebookLM / GPT / Gemini / Replit Agent
- n8n / Make.com / Zapier
- MCP servers / AI agent frameworks

### 範例（虛構）

> **最近我用 NotebookLM 來整理一份 50 頁的產業報告**。它的核心能力是把長文件變成可以對話的知識庫。對企業導入 AI 的人來說，這種工具代表「內部資料閱讀」這件事可以從一個小時壓到三分鐘。
>
> 過去做這件事我會把報告印出來、畫重點、做摘要 PPT。現在直接丟進 NotebookLM 問問題就好。我自己這半年 hands-on 的觀察是：知識整理工作的單位成本掉了 90%。
>
> 從經營管理的角度看，企業內部的標準作業手冊、過去專案紀錄、客戶投訴歷史這些「不知道怎麼用」的資產，都可以這樣翻新。最直接的價值是新人訓練從 3 個月縮到 3 週。
>
> 不要急著用它取代人寫的洞察，但取代「找資料」這一段絕對划算。
>
> —
>

> Google Workspace Updates — NotebookLM
> https://example.com/notebooklm-blog

## 📰 F-DAILY-NEWS：新聞短評（二四六用）

**核心鉤子**：時效性事件 + 翻譯員視角 + hands-on 證據
**最吃的平台**：FB Page
**字數**：350-400
**用途**：每週 Tue/Thu/Sat 21:00 — 抓過去 24-30h 一條 AI 新聞，翻譯為企業老闆能用的話

### 結構（4 段）

```
[段 1 — 一句事實切入 + 戰略點題]
**[今天/昨天]，[新聞主角] 推出/發布/宣布 [事件]**。看起來只是又一個 AI 更新，但對 [受眾] 來說，這代表一件事：[一句戰略含義]。

[段 2 — hands-on 視角解釋]
我自己這半年也在 [hands-on 經驗]，最有感的是 [具體經驗 + 量化]。

[段 3 — 對企業 / 受眾的真實影響]
從經營管理的角度看，這對 [行業] 最直接的影響是 [可量化的變化]。[補充行業案例 + 30% / 6 個月 等量化]。

[段尾 — 結論型]
[一句行動建議或長期觀察]。

—

資料來源：
[Source name] — URL
```

### 題材關鍵字

routine 蒐集時優先順序：
- agent / Claude Code skill / MCP（Model Context Protocol）/ Claude / vibe coding
- NotebookLM / OpenAI / Anthropic / GPT / ChatGPT / Gemini
- AI agent 開源工具 / GitHub trending（前述關鍵字）

**篩選標準**：
- ✅ 過去 24-30h 內發布
- ✅ 普通人 / 企業老闆會聽到 / 可能用到的工具或事件
- ✅ 有「企業導入 / 經營決策」可解讀的角度
- ❌ 排除：純 benchmark 數字之爭、純人事任命、純股價變動、純財報
- ❌ 排除：超過 48 小時的舊聞

### 範例（虛構）

> **昨天 Anthropic 推出 Claude Code skills**。看起來只是給工程師多一個玩具，但對企業導入 AI 的人來說，這代表一件事：內部 SOP 跟 prompt 工程要合併寫了。
>
> 我自己這半年也在用 AI 寫程式，最有感的是：以前寫 SOP 是給新人看，寫 prompt 是給 AI 用，是兩件事。Skills 把它合併了 — 一份 markdown 同時是員工手冊跟 AI 操作手冊。
>
> 從經營管理的角度看，這對知識密集的服務業最直接的影響是：內部 know-how 累積方式要改寫。寫 SOP 的人未來不只是寫給新人看，是寫給 AI 看。文件的標準要重定，不然 AI 越升級，落差越大。
>
> 公司的 SOP 該重寫了，而且要用 AI 看得懂的格式。
>

## 📊 F-WEEK-RECAP：本週大事統整（週日用）

**核心鉤子**：週度框架 + 5-7 趨勢併陳 + 翻譯員 voice
**最吃的平台**：FB Page
**字數**：**800-1000**
**用途**：每週日 21:00 — 把本週 5-7 條 AI 大事整合成一篇深度週報

### 結構（5-7 段 + 資料來源）

```
[段 1 — 週度框架點題（80-120 字）]
本週（YYYY-MM-DD ~ MM-DD），AI 圈的 [n] 件大事，都指向一個共同方向：[本週核心命題]。
從經營管理的角度看，這幾條動態擺在一起，對 [受眾] 最值得思考的是 [整週解讀切入點]。

[段 2 — 趨勢 1（130-160 字）]
**[趨勢 1 標題]**
[一句事實 + 來源]。
從經營管理的角度看 / 我自己 hands-on 的觀察是，[戰略含義 + 對企業影響 + 量化或案例]。

[段 3-6 — 趨勢 2/3/4/5/6]（同上結構，每段 130-160 字）

[段尾 — 整週 takeaway（80-120 字）]
本週這幾件事擺在一起看，最值得企業思考的是 [核心 takeaway]。
下一週可以 [建議行動方向]。

—

資料來源：
[Source 1] — URL
[Source 2] — URL
[Source 3] — URL
（共 5-7 條）
```

### 字數分配

| 段落 | 字數 |
|---|---|
| 段 1（框架）| 80-120 |
| 段 2-6（5 個趨勢）| 5 × 130-160 = 650-800 |
| 段尾（takeaway）| 80-120 |
| **總計** | **810-1040** |

### 為什麼週日 800-1000 而非 600

- 受眾預期週日有空閒消化深度內容
- 5-7 條 vs 1 條 → 廣度優先（讓讀者一篇看完一週）
- FB 演算法會把長 dwell time 內容判定為高品質
- 但**仍守翻譯員 voice** → 不破壞品牌一致性

### 範例（虛構，節錄第 1 段 + 第 1 個趨勢段）

> 本週（2026-XX-XX ~ XX-XX），AI 圈的 6 件大事，都指向一個共同方向：基礎設施標準化加速。從經營管理的角度看，這幾條動態擺在一起，對中小企業老闆最值得思考的是 — 過去 6 個月觀望的「要不要導入 AI」問題，現在變成「導入哪一條主線最划算」。
>
> **MCP 進入 Linux Foundation 主導的開源基金會**
> 4 月底，MCP 正式進駐 Linux Foundation 旗下新成立的 AI 開源基金會，與多個 agent framework 同列。這意味著 MCP 從某廠商的私有規範，變成跨廠商的開源標準。從經營管理的角度看，企業在採購 AI 工具時可以更放心 — 跟得上 MCP 標準的廠商基本上不會被綁定到單一模型廠，這對長期總成本控制是好消息。

## 🛡️ Voice Lock 8 條（三模式統一）

任何草稿生成後必須對照（詳見 `voice_lock.md`）：

1. **字數**：A 系列 350-400；C 系列 800-1000
2. **段落**：A 系列 4 段；C 系列 5-7 段（含資料來源前的本文）
3. **第 1 段**：粗體開頭錨點 + 戰略點題
4. **戰略段**：含「**從經營管理的角度看**」（或同義句）
5. **無 jargon**：避開 9 個 jargon 詞（見 `translation_dict.md`）
6. **emoji = 0**
7. **驚嘆號 ≤ 1**（最好 = 0）
8. **收尾**：結論型，不問問題、不召集留言、無 hashtag

任一條 ❌ → 重生成。
